Adapter onboarding

Onboard a target adapter — REST · MCP · A2A

The system under test joins the simulator through an adapter: a small declarative spec that exposes send a message, reset session, and (optional) create session. Pick a transport below — onboarding produces a versioned adapter record and runs a one-prompt smoke test before commit.

3 transports smoke-tested versioned per edit

01Identity

A human-readable name for this adapter and the base URL it lives at.

Shown in run lists and adapter cards.
All operations.*.path values are joined onto this.

02Auth

Pick an auth scheme. Credentials are referenced — never inlined.

A pointer to where the secret lives (vault, env, secret manager).
Sent on every request, in addition to auth headers.

03Operations

Three operations Simulation actually calls. Use Jinja-style placeholders {{ session_id }}, {{ message }}, {{ persona_id }}; extract responses with JSONPath.

POST send_message required
POST reset_session optional
If absent, Simulation falls back to rotating session_id locally.
POST create_session optional
Some adapters require explicit session creation; if absent, the SUT must accept new IDs as fresh.
